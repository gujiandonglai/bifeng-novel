# TXT → 网站 自动发布工具

`tools/txt_to_site.py` 用来把 `novels_txt/` 里的 `.txt` 小说批量转换为笔锋网站可以直接读取的数据（`books/<id>/` + `data/books.json`），全程无需手动拆章节、无需手动写 JSON。

## 用法

```bash
# 1. 把要发布的 txt 放进 novels_txt/ 目录
cp 我的小说.txt novels_txt/

# 2. 在项目根目录运行（需要 Python 3，不需要装任何第三方库）
python3 tools/txt_to_site.py

# 3. 运行结束后，books/ 和 data/books.json 已经更新好了
#    直接把整个项目目录推送到 GitHub 即可部署
```

可选参数：

```bash
python3 tools/txt_to_site.py --root /path/to/项目 --author "你的笔名"
```

## 它是怎么识别章节的

- **卷名**：形如「第一卷」「第二部」「第一阶段」「番外篇」等，独占一行
- **章节名**：形如「第1章」「第十章：标题」「第3回」「第5话」，
  以及「楔子」「序章」「尾声」「终章」「番外」等独立关键词标题
- 两者都要求整行较短（≤60 字）且格式匹配，避免把正文里长句子里
  偶然出现的「第二天……」误判成标题
- 章节文件按识别顺序生成 `0001.md`、`0002.md`……卷名信息会一并写入
  `book.json`，网站的目录页和阅读器侧边栏会自动按卷分组展示

如果一本书完全没有识别到任何标题格式，工具会把全文当成单章处理，
不会报错中断，但建议人工检查一下原始 txt 的排版。

## 排版假设

工具假设 txt 是「一行 = 一个自然段，段落之间空行分隔」这种最常见的
网文 txt 排版。如果你的文件是长段落被强制换行（而不是用空行分段），
转换后可能会被拆得更碎，建议转换后抽查一两章效果，必要时手动调整
对应的 `.md` 文件。

## 哪些字段是自动生成的，哪些会被保留

| 字段 | 每次运行 | 说明 |
|---|---|---|
| `chapters` / `words` / `status` | 总是重新生成 | 直接由 txt 正文推导，保证和源文件同步 |
| `update` | 总是重新生成 | 取 txt 文件的最后修改时间 |
| `author` / `category` / `cover` / `intro` / `tags` / `rating` | 首次生成后保留 | 这些字段第一次运行时会给出合理默认值（作者用 `--author` 指定、分类是关键词猜测、封面是占位 SVG、简介是正文前 120 字），后续如果你在 `book.json` 里手动改过，重复运行**不会覆盖**你的修改 |
| `views` / `recommend`（仅 `data/books.json`） | 保留 | 首次为 0 / false，之后按 `data/books.json` 里已有的值保留 |

也就是说：第一次运行拿到的是「能用但比较粗糙」的默认值，你可以在
`book.json` 里把分类、简介、封面换成更准确的内容；之后哪怕这本书
又更新了新章节，重新运行工具也只会追加/刷新章节数据，不会把你
手动润色过的部分冲掉。

## 分类是怎么猜的

工具会用一份关键词词典（修炼/仙/法宝 → 玄幻，总裁/直播/公司 → 都市，
江湖/武功 → 武侠……）对正文前 4000 字做关键词计数，取命中最多的分类；
全部没命中则退回默认分类。**这只是权重很低的粗略提示**，不是真正的
语义理解，建议每本新书都人工确认一下分类是否准确。

## 目前的局限

- 不做 OCR / 图片封面识别，封面统一是自动生成的渐变色占位 SVG，
  想用真实封面直接替换 `books/<id>/cover.jpg` 并在 `book.json` 里把
  `"cover"` 改成对应文件名即可
- 简介是从正文前 120 字截取的，不是真正的内容摘要，建议替换成
  你自己写的简介
- 分类识别、书名识别都是规则+关键词方式，不涉及任何联网调用，
  遇到排版特别不规范的 txt 可能需要人工兜底
